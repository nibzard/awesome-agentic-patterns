## Executive Summary

The **Deterministic Security Scanning Build Loop** pattern addresses a critical gap in AI-assisted code generation: ensuring security validation through deterministic, automated build tools rather than relying on non-deterministic LLM behavior. The pattern implements a two-phase approach where code is first generated by an LLM, then validated by deterministic security scanning tools that provide binary pass/fail signals with structured error feedback.

**Key Finding:** This pattern represents a convergence of three well-established research areas:
1. **Reproducible/Deterministic Builds** for supply chain security
2. **Static Analysis (SAST)** integrated into CI/CD pipelines
3. **LLM-based Code Generation** with security verification

While no single academic paper explicitly describes this exact pattern, the theoretical foundations are well-established across multiple research communities (2018-2025).

## 4. Technical Architecture

### 4.1 How the Pattern Works (The Mechanism)

The pattern implements a **two-phase approach**:

1. **Phase 1 (Generation):** Non-deterministic LLM generates code based on context
2. **Phase 2 (Backpressure):** Deterministic security scanning tools validate the code with binary pass/fail signals

**Key Insight:** Security requires absolute determinism—code is either secure or not. Non-deterministic approaches (Cursor rules, MCP security tools) are merely suggestions that may or may not be followed.

### 4.2 Key Components and Interactions

```
[Agent] --generates--> [Code] --triggers--> [Build Target]
                                         |
                                         v
[Security Tools] --Semgrep/Bandit/SonarQube--> [Exit Code]
                                         |
                                         v
[Structured Error JSON] --feedback--> [Agent] --fixes--> [Code]
```

The pattern integrates existing security scanning tools (SAST, DAST, PBT) directly into the build target using:
- Makefiles
- package.json scripts
- CI/CD pipelines

### 4.3 Data Flow and Control Flow

| Aspect | Flow |
|--------|------|
| **Control Flow** | Generate → Build → Scan → (Pass/Complete) OR (Fail/Regenerate) |
| **Data Flow** | Code changes trigger build → Security tools analyze → Exit codes + JSON output → Agent parses and fixes |
| **Feedback Type** | Binary (pass/fail) with structured error details |
| **Termination** | Only when all security checks pass |

### 4.4 Integration Points with CI/CD Systems

The pattern operates in two loops:

1. **Inner Loop (Development):** Local fast feedback after each code change
2. **Outer Loop (CI/CD):** Final validation before merge

A unified rules database ensures consistency between local and CI environments.

### 4.5 Benefits vs Trade-offs

| Aspect | Deterministic Build Loop | Non-Deterministic Suggestions |
|--------|--------------------------|-------------------------------|
| **Guarantee** | Binary pass/fail enforced | Best-effort compliance |
| **Reliability** | 100% (tool determinism) | Variable (LLM randomness) |
| **Coverage** | All code paths | LLM-dependent |
| **Iteration** | Automated retry loop | Manual intervention |

**Benefits:**
- 100% deterministic security validation
- Leverages battle-tested tools (Semgrep, Bandit, SonarQube)
- Works with any coding agent/harness
- Unified security rules across inner/outer loops
- 2-5x faster agent iteration with structured errors

**Trade-offs:**
- Increased build time (5-30 seconds per scan)
- False positives requiring rule tuning
- CI resource usage for scans
- Tool maintenance overhead
